# 4. 비기능 요구사항

## NFR-001: 성능

| 항목 | 내용 |
|------|------|
| **ID** | NFR-001 |
| **우선순위** | Medium |

### 페이지 로드 시간

| 측정 항목 | 목표 |
|-----------|------|
| 초기 페이지 로드 (LCP) | 3초 이내 |
| 페이지 간 이동 (SPA 라우팅) | 1초 이내 |
| 데이터 테이블 렌더링 | 2초 이내 |

### API 응답 시간

| 항목 | 목표 |
|------|------|
| 목록 조회 API | 2초 이내 |
| 단일 항목 조회 API | 1초 이내 |
| 생성/수정/삭제 API | 3초 이내 |
| 이미지 업로드 | 10초 이내 (파일 크기에 따라 상이) |
| 블록체인 배포 요청 | 비동기 처리 (응답 즉시, 결과는 폴링) |

### 구현 방식

- TanStack React Query의 캐싱으로 반복 요청 최소화
- `staleTime` 설정으로 불필요한 리페치 방지
- Next.js App Router의 서버 컴포넌트 활용
- 동적 import(`next/dynamic`)로 대형 컴포넌트 코드 스플리팅

## NFR-002: 보안

| 항목 | 내용 |
|------|------|
| **ID** | NFR-002 |
| **우선순위** | High |

### 인증/인가

| 항목 | 구현 방식 |
|------|-----------|
| 토큰 방식 | JWT Bearer Token (HS256) |
| Access Token 만료 | 24시간 |
| Refresh Token 만료 | 7일 |
| 보호된 라우트 | 미인증 시 `/login`으로 리다이렉트 |
| API 권한 | 모든 `/admin/**` 요청에 Bearer 토큰 필수 |

### 통신 보안

- **HTTPS** 전용 통신 (HTTP → HTTPS 자동 업그레이드)
- API Base URL은 환경변수(`NEXT_PUBLIC_API_URL`)로 관리
- 민감 정보(비밀번호, 토큰)는 응답 로그에 노출 금지

### CORS

- 백엔드에서 허용된 Origin만 API 접근 가능
- 개발 환경: `localhost:3000` 허용

### XSS 방지

- React의 기본 이스케이프 처리 활용
- `dangerouslySetInnerHTML` 미사용 원칙
- 사용자 입력값은 서버로 전송 전 trim 처리

### 입력값 검증

- 이메일 형식 검증 (프론트엔드)
- 비밀번호 최소 길이 검증 (최소 6자)
- 필수 필드 누락 시 제출 차단

## NFR-003: 사용성

| 항목 | 내용 |
|------|------|
| **ID** | NFR-003 |
| **우선순위** | Medium |

### 반응형 디자인

| 브레이크포인트 | 레이아웃 |
|----------------|----------|
| 모바일 (< 768px) | 사이드바 숨김, 햄버거 메뉴 |
| 태블릿 (768px ~ 1024px) | 사이드바 축소 또는 오버레이 |
| 데스크탑 (> 1024px) | 사이드바 고정 표시 |

### 다크 모드

- `next-themes` 기반 라이트/다크 테마 전환
- 시스템 테마 자동 감지
- 테마 선택은 로컬 스토리지에 저장

### 피드백 및 UX

| 상황 | 처리 방식 |
|------|-----------|
| 데이터 로딩 중 | 스피너 또는 스켈레톤 표시 |
| API 성공 | 토스트 알림 (녹색) |
| API 실패 | 토스트 알림 (빨간색) + 에러 메시지 |
| 빈 데이터 | EmptyState 컴포넌트 표시 |
| 삭제/강제 종료 | 확인 다이얼로그 표시 후 진행 |

### 언어

- UI 레이블 및 메시지: 한국어
- 기술 용어(API, JWT, CRUD, UUID 등): 영문 그대로 사용
- 날짜/시간: ISO 8601 형식 표시, 한국 시간(KST, +09:00)

## NFR-005: 가용성 및 에러 처리

| 항목 | 내용 |
|------|------|
| **ID** | NFR-005 |
| **우선순위** | Medium |

### 백엔드 의존성

- Admin Dashboard는 백엔드 API에 전적으로 의존
- 백엔드 다운 시 API 에러를 사용자에게 명확히 표시
- 일부 데이터는 React Query 캐시를 통해 오프라인 상태에서 표시 가능

### 에러 처리 전략

| 에러 유형 | 처리 방식 |
|-----------|-----------|
| 401 Unauthorized | 토큰 갱신 시도 → 실패 시 로그아웃 |
| 403 Forbidden | 권한 없음 메시지 표시 |
| 404 Not Found | 빈 상태 또는 "항목을 찾을 수 없습니다" 표시 |
| 400 Bad Request | 에러 메시지를 폼 필드 또는 토스트로 표시 |
| 500 Internal Server Error | "서버 오류가 발생했습니다. 잠시 후 다시 시도해 주세요" 표시 |
| 네트워크 오류 | "네트워크 연결을 확인해 주세요" 표시 |

### API 클라이언트 에러 처리

- 모든 API 호출은 `lib/api/client.ts`의 공통 에러 핸들러를 통과
- HTTP 상태코드별 공통 처리 로직 적용
- React Query의 `onError` 콜백으로 토스트 알림 표시
