# 사용자 관리 테스트 케이스

## TC-USER-001: 사용자 목록 테이블 렌더링

- **우선순위**: High
- **유형**: 기능
- **사전조건**: ADMIN 계정으로 로그인됨. 사용자 데이터가 존재함
- **테스트 절차**:
  1. `/users` 페이지 접속
  2. 페이지 로드 완료 대기
- **기대결과**: 사용자 목록이 테이블로 표시됨. 컬럼: 사용자명, 이메일, 이름, 역할(Badge), 상태(색상 점), 가입일. 상단에 총 사용자/활성/정지/비활성 통계 카드 4개가 표시됨
- **관련 API**: `GET /admin/users`

---

## TC-USER-002: 사용자 검색 및 필터링

- **우선순위**: High
- **유형**: 기능
- **사전조건**: `/users` 페이지 접속됨. 여러 사용자가 존재함
- **테스트 절차**:
  1. 검색창에 특정 이메일 또는 닉네임 입력
  2. 역할 필터에서 "ADMIN" 선택
  3. 상태 필터에서 "ACTIVE" 선택
  4. 필터 적용 확인
- **기대결과**: 각 필터 조건에 맞는 사용자만 테이블에 표시됨. 필터 변경 시 페이지가 0으로 리셋됨
- **관련 API**: `GET /admin/users` (파라미터: search, role, status)

---

## TC-USER-003: 사용자 생성 성공

- **우선순위**: High
- **유형**: 기능
- **사전조건**: `/users` 페이지 접속됨
- **테스트 절차**:
  1. "사용자 추가" 버튼 클릭
  2. 사용자 생성 다이얼로그에서 이메일, 비밀번호(6자 이상), 닉네임 입력
  3. 역할을 "USER"로 선택
  4. 저장 버튼 클릭
- **기대결과**: 다이얼로그가 닫히고 성공 Toast 알림이 표시됨. 사용자 목록이 갱신되어 새 사용자가 나타남
- **관련 API**: `POST /admin/users`

---

## TC-USER-004: 중복 이메일로 사용자 생성 실패

- **우선순위**: Medium
- **유형**: 에러처리
- **사전조건**: 이미 존재하는 이메일 주소를 알고 있음
- **테스트 절차**:
  1. "사용자 추가" 버튼 클릭
  2. 이미 존재하는 이메일 입력
  3. 비밀번호 및 닉네임 입력
  4. 저장 버튼 클릭
- **기대결과**: 에러 Toast 알림이 표시됨. 다이얼로그가 유지됨. 중복 이메일 관련 에러 메시지가 표시됨
- **관련 API**: `POST /admin/users` → 4xx 에러

## TC-USER-005: 필수 필드 누락 시 사용자 생성 실패

- **우선순위**: Medium
- **유형**: 에러처리
- **사전조건**: 사용자 생성 다이얼로그가 열려 있음
- **테스트 절차**:
  1. 이메일 필드를 비워둔 채 저장 버튼 클릭
  2. 비밀번호 필드를 비워둔 채 저장 버튼 클릭
- **기대결과**: HTML5 또는 커스텀 폼 유효성 검사에 의해 제출이 차단됨. 에러 메시지가 표시됨
- **관련 API**: 없음 (클라이언트사이드 유효성 검사)

## TC-USER-006: 사용자 정보 수정

- **우선순위**: High
- **유형**: 기능
- **사전조건**: 수정할 사용자가 존재함
- **테스트 절차**:
  1. 사용자 목록에서 특정 사용자의 액션 메뉴(점 3개) 클릭
  2. "수정" 선택
  3. 닉네임 또는 이메일 수정
  4. 저장 버튼 클릭
- **기대결과**: 성공 Toast 알림이 표시됨. 사용자 목록이 갱신되어 변경된 정보가 반영됨
- **관련 API**: `PUT /admin/users/{id}`

## TC-USER-007: 사용자 삭제 - 확인 다이얼로그 및 성공

- **우선순위**: High
- **유형**: 기능
- **사전조건**: 삭제할 사용자가 존재함
- **테스트 절차**:
  1. 사용자 목록에서 특정 사용자의 액션 메뉴 클릭
  2. "삭제" 선택
  3. 확인 다이얼로그에서 대상 이메일 확인
  4. "삭제" 버튼 클릭
- **기대결과**: 확인 다이얼로그에 대상 사용자 이메일이 표시됨. 삭제 성공 후 Toast 알림 표시. 사용자 목록에서 해당 사용자가 제거됨
- **관련 API**: `DELETE /admin/users/{id}`

## TC-USER-009: 역할 변경 (USER → ADMIN)

- **우선순위**: High
- **유형**: 기능
- **사전조건**: USER 역할의 사용자가 존재함
- **테스트 절차**:
  1. 사용자 목록에서 USER 역할 사용자의 액션 메뉴 클릭
  2. "역할 변경" 선택
  3. 역할 변경 다이얼로그에서 "ADMIN" 선택
  4. "변경" 버튼 클릭
- **기대결과**: 성공 Toast 알림이 표시됨. 사용자 목록이 갱신되어 역할이 ADMIN으로 변경됨
- **관련 API**: `PATCH /admin/users/role`

## TC-USER-010: 사용자 상세 다이얼로그 조회

- **우선순위**: Medium
- **유형**: UI
- **사전조건**: 사용자 데이터가 존재함
- **테스트 절차**:
  1. 사용자 목록에서 특정 사용자의 액션 메뉴 클릭

- **관련 API**: `GET /admin/users/{id}`

## TC-USER-012: 필터 초기화

- **우선순위**: Low
- **유형**: UI
- **사전조건**: 검색어 및 필터가 적용된 상태
- **테스트 절차**:
  1. 검색어와 역할 필터, 상태 필터를 모두 적용
  2. "초기화" 또는 리셋 버튼 클릭
- **기대결과**: 모든 필터가 초기화되고 전체 사용자 목록이 표시됨
- **관련 API**: `GET /admin/users`

## TC-USER-013: 페이지네이션 동작

- **우선순위**: Medium
- **유형**: 기능
- **사전조건**: 10명 이상의 사용자가 존재함 (페이지 크기: 10)
- **테스트 절차**:
  1. `/users` 페이지 접속
  2. 테이블 하단 페이지네이션에서 다음 페이지 클릭
- **기대결과**: 다음 10명의 사용자가 표시됨. 페이지 번호가 변경됨. 이전 페이지 버튼이 활성화됨
- **관련 API**: `GET /admin/users` (page, size 파라미터)
