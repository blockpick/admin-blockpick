# 게임 관리 테스트 케이스

## TC-GAME-001: 게임 목록 조회 및 페이지네이션

- **우선순위**: High
- **유형**: 기능
- **사전조건**: ADMIN 계정으로 로그인됨. 게임 데이터가 존재함
- **테스트 절차**:
  1. `/games` 페이지 접속
  2. 페이지 로드 완료 대기
  3. 테이블 하단 페이지네이션 확인
  4. 다음 페이지 버튼 클릭
- **기대결과**: 게임 목록이 테이블로 표시됨 (게임명, 유형, 카테고리, 상태, 보상포인트 컬럼). Spring Page 기반 페이지네이션 동작. 상단에 총/활성/예정/종료 통계 카드 4개 표시됨
- **관련 API**: `GET /admin/games` (page, size 파라미터)

---

## TC-GAME-002: 게임 유형 필터

- **우선순위**: High
- **유형**: 기능
- **사전조건**: DAILY, SELECT, VIBE, PRIME 유형의 게임이 각각 존재함
- **테스트 절차**:
  1. GameFilterBar에서 유형 필터를 "DAILY"로 선택
  2. 목록 갱신 확인
  3. 필터를 "SELECT"로 변경
- **기대결과**: 선택된 유형의 게임만 목록에 표시됨. 페이지가 0으로 리셋됨
- **관련 API**: `GET /admin/games` (type 파라미터)

## TC-GAME-003: 게임 상태 필터

- **우선순위**: High
- **유형**: 기능
- **사전조건**: 다양한 상태(SCHEDULED, IN_PROGRESS, ENDED 등)의 게임이 존재함

- **기대결과**: 선택된 상태의 게임만 표시됨
- **관련 API**: `GET /admin/games` (status 파라미터)

## TC-GAME-004: 게임 제목 검색

- **우선순위**: High
- **유형**: 기능
- **사전조건**: 제목을 알고 있는 게임이 존재함
- **테스트 절차**:

- **기대결과**: 해당 키워드가 포함된 게임만 표시됨

## TC-GAME-005: 게임 생성 성공 (필수 필드)

- **우선순위**: High
- **유형**: 기능
- **사전조건**: `/games` 페이지 접속됨
- **테스트 절차**:
  1. "게임 추가" 버튼 클릭
  2. 게임 생성 다이얼로그에서 필수 필드 입력: 제목, 유형(DAILY), 시작시간, 종료시간, 참가비, 화폐(CASH), 최소/최대 참가수
  3. 저장 버튼 클릭
- **기대결과**: 다이얼로그가 닫히고 성공 Toast 표시. 게임 목록이 갱신되어 새 게임이 나타남
- **관련 API**: `POST /admin/blockchain/games` (deployContract: true) 또는 일반 생성 API

---

## TC-GAME-006: 게임 생성 - 필수 필드 누락 시 유효성 검사

- **우선순위**: Medium
- **유형**: 에러처리
- **사전조건**: 게임 생성 다이얼로그가 열려 있음
- **테스트 절차**:
  1. 제목 필드를 비워둔 채 저장 버튼 클릭
  2. 시작/종료 시간 없이 저장 시도
- **기대결과**: 폼 유효성 검사에 의해 제출이 차단됨. 에러 메시지가 표시됨
- **관련 API**: 없음 (클라이언트사이드 유효성 검사)

## TC-GAME-007: 게임 수정

- **우선순위**: High
- **유형**: 기능
- **사전조건**: 수정할 게임이 존재함
- **테스트 절차**:
  1. 게임 목록에서 특정 게임의 액션 메뉴 클릭
  2. "수정" 선택
  3. 제목 또는 설명 변경
  4. 저장 버튼 클릭
- **기대결과**: 성공 Toast 표시. 게임 목록이 갱신되어 변경된 정보가 반영됨
- **관련 API**: `PUT /admin/games`

---

## TC-GAME-008: 게임 삭제 - 확인 다이얼로그 및 성공

- **우선순위**: High
- **유형**: 기능
- **사전조건**: 삭제할 게임이 존재함
- **테스트 절차**:
  1. 게임 액션 메뉴에서 "삭제" 선택
  2. 확인 다이얼로그에서 게임 제목 확인
  3. "삭제" 버튼 클릭
- **기대결과**: 확인 다이얼로그에 게임 제목이 표시됨. 삭제 성공 Toast 표시. 목록에서 제거됨
- **관련 API**: `DELETE /admin/games/{id}`

---

## TC-GAME-009: 게임 강제 종료 (IN_PROGRESS → ENDED)

- **우선순위**: High
- **유형**: 기능
- **사전조건**: IN_PROGRESS 상태의 게임이 존재함
- **테스트 절차**:
  1. IN_PROGRESS 상태 게임의 액션 메뉴 클릭
  2. "강제 종료" 선택 (상태가 ACTIVE 또는 IN_PROGRESS일 때만 표시됨)
  3. 확인 다이얼로그에서 "종료" 버튼 클릭
- **기대결과**: 성공 Toast 표시. 게임 상태가 ENDED로 변경됨
- **관련 API**: `PUT /admin/games/{id}/force-end`

---

## TC-GAME-010: 강제 종료 버튼 - ENDED 상태에서 미표시

- **우선순위**: Medium
- **유형**: UI
- **사전조건**: ENDED 상태의 게임이 존재함
- **테스트 절차**:
  1. ENDED 상태 게임의 액션 메뉴 열기
  2. 메뉴 항목 확인
- **기대결과**: "강제 종료" 메뉴 항목이 표시되지 않음. "정산" 메뉴가 표시됨
- **관련 API**: 없음 (UI 조건 분기)

## TC-GAME-011: 게임 정산 (ENDED → 당첨자 선정)

- **우선순위**: High
- **유형**: 기능
- **사전조건**: ENDED 상태의 게임이 존재함
- **테스트 절차**:
  1. ENDED 상태 게임의 액션 메뉴에서 "정산" 선택

- **관련 API**: `POST /admin/games/{id}/settle`

## TC-GAME-015: 빈 상태 표시

- **우선순위**: Low
- **유형**: UI
- **사전조건**: 필터를 적용했을 때 결과가 없는 조건
- **테스트 절차**:
  1. 게임이 존재하지 않는 유형이나 상태 필터 적용
- **기대결과**: EmptyState 컴포넌트가 표시됨: "게임이 없습니다" 메시지와 "게임 추가" 액션 버튼
- **관련 API**: `GET /admin/games` → 빈 결과
