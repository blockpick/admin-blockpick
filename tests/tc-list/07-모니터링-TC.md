# 시스템 모니터링 테스트 케이스

## TC-MON-001: 시스템 헬스 체크 - 정상 상태

- **우선순위**: High
- **유형**: 기능
- **사전조건**: ADMIN 계정으로 로그인됨
- **테스트 절차**:
  1. `/monitoring` 페이지 접속
  2. 페이지 로드 완료 대기
  3. 상단 헬스 카드 확인
- **기대결과**: 시스템 상태 카드에 "HEALTHY" 또는 "UP" 상태가 표시됨. 미발행 이벤트수, 대기 중 트랜잭션수, 실패한 트랜잭션수 카드가 표시됨
- **관련 API**: `GET /admin/monitoring/health`

---

## TC-MON-002: 시스템 헬스 체크 - DEGRADED 상태

- **우선순위**: High
- **유형**: 에러처리
- **사전조건**: 시스템에 이상 상태가 발생한 경우 (실패 트랜잭션 다수 등)
- **테스트 절차**:
  1. `/monitoring` 페이지 접속
  2. 헬스 상태 카드 확인
- **기대결과**: DEGRADED 상태일 때 AlertCircle 아이콘으로 경고 표시됨. alerts 배열의 CRITICAL/WARNING 메시지가 UI에 반영됨
- **관련 API**: `GET /admin/monitoring/health` → status: "DEGRADED"

---

## TC-MON-003: 시스템 헬스 조회 실패 시 에러 메시지 표시

- **우선순위**: Medium
- **유형**: 에러처리
- **사전조건**: 모니터링 API 서버가 응답하지 않는 상태
- **테스트 절차**:
  1. `/monitoring` 페이지 접속
  2. 상단 헬스 영역 확인
- **기대결과**: "시스템 상태를 불러올 수 없습니다. 서버 오류가 발생했습니다." 에러 메시지가 빨간색 경고 박스로 표시됨
- **관련 API**: `GET /admin/monitoring/health` → 오류

---

## TC-MON-004: 이벤트 발행 상태 조회

- **우선순위**: High
- **유형**: 기능
- **사전조건**: `/monitoring` 페이지 접속됨. 미발행 이벤트가 존재함
- **테스트 절차**:
  1. "이벤트" 탭 클릭
  2. 미발행 이벤트 목록 확인
- **기대결과**: 미발행 이벤트 수 카드와 최근 미발행 이벤트 목록이 표시됨. 각 이벤트에 eventType, aggregateType, aggregateId, occurredAt, 발행여부 Badge가 표시됨
- **관련 API**: `GET /admin/monitoring/events`

---

## TC-MON-005: 이벤트 재발행

- **우선순위**: High
- **유형**: 기능
- **사전조건**: 미발행 이벤트가 존재함
- **테스트 절차**:
  1. "이벤트" 탭에서 미발행 이벤트 확인
  2. 재발행 버튼 클릭 (구현된 경우)
- **기대결과**: 성공 Toast("이벤트 재시도가 요청되었습니다") 표시됨. 이벤트 상태가 갱신됨

> ⚠️ 미구현: 현재 이벤트 목록에 재발행 버튼 UI가 없음. `handleRetryEvent` 함수는 구현되어 있으나 호출 UI가 없음
- **관련 API**: `POST /admin/monitoring/events/{eventId}/retry`

---

## TC-MON-006: 트랜잭션 처리 상태 조회

- **우선순위**: High
- **유형**: 기능
- **사전조건**: 블록체인 트랜잭션이 존재함
- **테스트 절차**:
  1. "트랜잭션" 탭 클릭
  2. 상태별 카운트 및 최근 실패 트랜잭션 확인
- **기대결과**: PENDING/SENT/CONFIRMED/FAILED/REFUNDED 각 상태별 카운트 카드가 표시됨. 최근 실패 트랜잭션 목록에 intentId, entryId, 에러코드, 에러메시지, 실패시간이 표시됨
- **관련 API**: `GET /admin/monitoring/transactions`

---

## TC-MON-007: 시스템 로그 조회

- **우선순위**: Medium
- **유형**: 기능
- **사전조건**: 시스템 로그가 존재함
- **테스트 절차**:
  1. "로그" 탭 클릭 (기본 탭)
  2. 로그 목록 확인
- **기대결과**: 시간, 레벨(Badge), 서비스(Badge), 메시지 컬럼으로 로그가 표시됨
- **관련 API**: 로그 API (미구현 가능성 있음)

---

## TC-MON-008: 로그 레벨 필터

- **우선순위**: Medium
- **유형**: 기능
- **사전조건**: 로그 탭이 활성화된 상태
- **테스트 절차**:
  1. 레벨 선택 드롭다운에서 "ERROR" 선택
  2. 결과 확인
- **기대결과**: ERROR 레벨의 로그만 표시됨
- **관련 API**: 로그 API (level 파라미터)

---

## TC-MON-009: 로그 서비스 필터

- **우선순위**: Medium
- **유형**: 기능
- **사전조건**: 로그 탭이 활성화된 상태
- **테스트 절차**:
  1. 서비스 선택 드롭다운에서 특정 서비스(예: "blockchain") 선택
  2. 결과 확인
- **기대결과**: 선택한 서비스의 로그만 표시됨
- **관련 API**: 로그 API (service 파라미터)

---

## TC-MON-010: 메트릭 차트 조회

- **우선순위**: Low
- **유형**: 기능
- **사전조건**: 메트릭 데이터가 존재함
- **테스트 절차**:
  1. "메트릭" 탭 클릭
  2. 기간 선택에서 "일", "주", "월" 각각 선택
- **기대결과**: CPU 사용률, 메모리 사용률, 요청 수 차트가 표시됨. 기간 변경 시 차트 데이터가 갱신됨
- **관련 API**: 메트릭 API (interval 파라미터)

> ⚠️ 미구현: 메트릭 API 및 로그 API가 ADMIN_API_SPEC에 정의되지 않음. 별도 구현 필요
